Frontend Practical Exam — React ​

ระบบ: Fixly
เวลา: 5 ชั่วโมง
คะแนนเต็ม: 100 คะแนน

1. เป้าหมาย ​

สร้าง Frontend สำหรับระบบแจ้งปัญหาและติดตามงานซ่อม จาก UI reference และ API contract ที่ได้รับ โดยเชื่อมกับ be-exam API จริง (ต้อง login เพื่อขอ token)

Live URLs

  • เว็บตัวอย่าง (UI reference): https://fixly-frontend-jvxs.onrender.com
  • API base (be-exam ที่ deploy ไว้): https://fixly-backend-6pzm.onrender.com/api

ข้อสอบวัด:

  • การอ่าน UI
  • Component design
  • State และ form
  • Authentication / login + การแนบ token
  • API integration
  • Validation
  • Loading/error/empty state
  • Responsive layout

2. Technology ​

ต้องใช้:

  • React
  • Vite
  • JavaScript เท่านั้น
  • .js และ .jsx

ห้าม:

  • TypeScript
  • .ts, .tsx
  • Next.js
  • Hardcode ข้อมูลแทน API
  • เปลี่ยน API contract
  • ใช้ AI

3. งานที่ต้องทำ ​

FE-00 Login ​

  • ฟอร์ม username/password
  • ยิง POST /auth/login ของ be-exam เพื่อขอ JWT
  • เก็บ token และแนบ Authorization: Bearer <token> ทุก request
  • เก็บ role ของผู้ใช้ จากผลลัพธ์ login เพื่อปรับ UI ตามสิทธิ์
  • จัดการ invalid credentials / auth error
  • กันเข้าหน้าอื่นเมื่อยังไม่ login + รองรับ logout
  • ใช้ user ที่ seed ไว้ใน be-exam (admin/admin123 = admin, demo/demo1234 = user)

FE-01 Ticket List ​

แสดง:

  • Title
  • Category
  • Location
  • Reporter
  • Assignee (ผู้รับผิดชอบ)
  • Priority
  • Status
  • Edit
  • Change status

USER เห็นเฉพาะ ticket ของตน, ADMIN เห็นทั้งหมด (backend แยกให้แล้ว — frontend ไม่ต้องส่ง filter เพิ่ม)

ต้องมี:

  • Loading
  • Error + retry
  • Empty state
  • Responsive desktop/mobile

FE-02 Search and Filters ​

ต้องมี:

  • Search
  • Category filter
  • Priority filter
  • Status filter
  • Clear filters

FE-03 Create Ticket (admin only) ​

  • เข้าถึง/แสดงได้เฉพาะ admin (ซ่อนปุ่ม "New ticket" และกันเข้าหน้าเมื่อไม่ใช่ admin)
  • ดึงรายชื่อผู้ใช้จาก GET /users มาทำ dropdown เลือกผู้รับผิดชอบ

Form:

  • title
  • description
  • reporterName
  • location
  • categoryId
  • assignedToId (ผู้รับผิดชอบ — required)
  • priority

Validation:

  • required (รวม responsible user)
  • title อย่างน้อย 3 ตัวอักษร
  • description อย่างน้อย 10 ตัวอักษร
  • reporterName อย่างน้อย 2 ตัวอักษร
  • location อย่างน้อย 2 ตัวอักษร

หลังสร้างสำเร็จ ต้องเห็นรายการใหม่โดยไม่ต้อง refresh เอง

FE-04 Edit Ticket ​

  • โหลดข้อมูลเดิม
  • แก้ไขและบันทึก
  • admin เปลี่ยนผู้รับผิดชอบ (reassign) ได้; USER แก้ได้เฉพาะ ticket ของตนและไม่มีช่องเลือกผู้รับผิดชอบ
  • จัดการ not found
  • update UI หลังสำเร็จ

FE-05 Change Status ​

  • เปลี่ยน status ผ่าน API
  • มี loading/disabled ระหว่าง request
  • update UI หลังสำเร็จ

FE-06 Error Handling ​

รองรับ:

  • VALIDATION_ERROR
  • UNAUTHORIZED / INVALID_CREDENTIALS (token หมดอายุ/login ผิด)
  • FORBIDDEN (ไม่มีสิทธิ์ เช่น ไม่ใช่ admin)
  • CATEGORY_NOT_FOUND
  • CATEGORY_INACTIVE
  • TICKET_NOT_FOUND / USER_NOT_FOUND
  • network error

FE-07 Summary ​

แสดง:

  • Open
  • In progress
  • Resolved
  • Active categories

4. UI Expectations ​

  • ไม่ต้อง pixel-perfect 100%
  • hierarchy อ่านง่าย
  • spacing สม่ำเสมอ
  • form label ชัดเจน
  • mobile 375px ใช้งานได้
  • status และ priority ต้องมีข้อความ ไม่ใช้สีอย่างเดียว
  • ไม่มี console error สำคัญ

6. เกณฑ์การให้คะแนนของ AI ​

การตรวจข้อสอบใช้ AI ช่วยประเมิน ผลงานตาม มิติคุณภาพ ด้านล่าง (ภาพรวม ไม่ใช่การไล่นับคะแนนรายข้อ) เพื่อความโปร่งใส ผู้เข้าสอบควรทำให้ครบทุกมิติ

  • ความครบของฟีเจอร์ — ทำงานได้จริงครบตาม FE-00 ถึง FE-07 (login, list, search/filter, create, edit, change status, summary)
  • Authentication และ role — แนบ Authorization: Bearer <token> ทุก request, ปรับ UI ตาม role, กันเข้าหน้า/ซ่อนปุ่มตามสิทธิ์, จัดการ logout และ token หมดอายุ
  • API integration — เชื่อม be-exam API จริง ไม่ hardcode ข้อมูล และไม่เปลี่ยน contract
  • Form และ validation — validate ครบตามกติกา (เช่น title ≥ 3, description ≥ 10), บังคับ required, แสดงข้อความ error ราย field ชัดเจน
  • State handling — มี loading / error + retry / empty state และอัปเดต UI หลังสร้าง/แก้ไข/เปลี่ยนสถานะโดยไม่ต้อง refresh เอง
  • Error handling — รองรับ error code ตามข้อ FE-06 และ network error ด้วยข้อความที่ผู้ใช้เข้าใจ
  • UI/UX — hierarchy อ่านง่าย, spacing สม่ำเสมอ, ใช้งานบน mobile 375px ได้, status/priority มีข้อความ (ไม่ใช้สีอย่างเดียว), ไม่มี console error สำคัญ
  • คุณภาพโค้ด — โครงสร้าง component อ่านและดูแลง่าย ใช้ JavaScript (.js/.jsx) ตามข้อกำหนด

Common Exam Policy ​

1. รูปแบบการสอบ ​

ข้อสอบแบ่งเป็น 2 ส่วนและสอบแยกกัน:

  • Frontend: React
  • Backend: Express + Prisma + PostgreSQL

ระยะเวลาสอบส่วนละ 5 ชั่วโมง

2. สิ่งที่อนุญาต ​

ผู้เข้าสอบสามารถ:

  • เปิดดูโปรเจกต์เก่าที่ตนเองเคยทำ
  • เปิดดู source code ของตนเอง
  • เปิดเอกสารทางการของ React, Express, Prisma, PostgreSQL และ JavaScript
  • ค้นหา syntax หรือ error message ผ่าน search engine
  • ใช้ editor, terminal, database client และ browser developer tools
  • ใช้จอเสริมได้ตามระเบียบของผู้คุมสอบ โดยทุกจอที่ใช้งานต้องสามารถตรวจสอบได้

3. สิ่งที่ห้าม ​

ห้ามใช้ AI ทุกประเภทระหว่างสอบ เช่น:

  • ChatGPT
  • Claude
  • Gemini
  • GitHub Copilot
  • Cursor AI
  • Windsurf
  • Codeium
  • Tabnine
  • AI extension หรือ AI autocomplete
  • AI search หรือ AI browser assistant
  • การให้ผู้อื่นช่วยเขียนหรือแก้โค้ด

ผู้เข้าสอบต้องปิดหรือ disable AI extension ก่อนเริ่มสอบ

4. การแชร์หน้าจอ ​

  • ต้องแชร์หน้าจอผ่าน Zoom ตลอดการสอบ
  • ต้องแชร์หน้าจอที่กำลังใช้งานจริง

  • ห้ามหยุดแชร์หน้าจอโดยไม่ได้รับอนุญาต

  • ผู้คุมสอบอาจขอตรวจ browser tab, extension, process และ terminal history

Domain Specification: Fixly ​

1. ภาพรวมระบบ ​

Fixly คือระบบแจ้งปัญหาและติดตามงานซ่อมภายในบริษัท

ตัวอย่างปัญหา:

  • เครื่องปรับอากาศไม่เย็น
  • หลอดไฟเสีย
  • คอมพิวเตอร์เปิดไม่ติด
  • อินเทอร์เน็ตใช้งานไม่ได้
  • โต๊ะหรือเก้าอี้ชำรุด

ผู้ใช้ต้องสามารถ:

  • เข้าสู่ระบบ (login)
  • ดูรายการแจ้งซ่อม
  • ค้นหาและกรองรายการ
  • สร้างรายการแจ้งซ่อม
  • แก้ไขข้อมูล
  • เปลี่ยนสถานะงาน
  • ดูข้อมูลสรุป

Frontend เชื่อมกับ be-exam API จริง (deploy ไว้ที่ https://fixly-backend-6pzm.onrender.com/api) ซึ่งบังคับ authentication — ผู้สอบต้องทำ login (username/password → JWT) เอง แล้วแนบ Authorization: Bearer <token> ทุก request

ผู้ใช้มี บทบาท (role) สองแบบ:

  • USER — เห็นเฉพาะ ticket ที่ถูกมอบหมายให้ตน, แก้ไข/เปลี่ยนสถานะได้เฉพาะ ticket ของตน, สร้าง ticket ไม่ได้
  • ADMIN — เห็น ticket ทั้งหมด และ สร้าง ticket พร้อมเลือกผู้รับผิดชอบ (assignee) ได้

UI ต้องปรับตาม role ที่ได้จากผลลัพธ์ login (เช่น ซ่อนปุ่ม "New ticket" เมื่อไม่ใช่ admin)

3. Business Rules ​

BR-01 Category ​

  • Ticket ต้องอ้างถึง Category ที่มีอยู่จริง
  • สร้างหรือแก้ Ticket ให้ใช้ได้เฉพาะ Category ที่ isActive = true

BR-02 Validation ​

  • title: 3–100 ตัวอักษร
  • description: 10–1000 ตัวอักษร
  • reporterName: 2–80 ตัวอักษร
  • location: 2–100 ตัวอักษร
  • priority: ต้องเป็นค่าที่กำหนด
  • status: ต้องเป็นค่าที่กำหนด
  • categoryId: ต้องเป็น integer และมีอยู่จริง

BR-03 Default values ​

เมื่อสร้าง Ticket:

  • หากไม่ส่ง priority ให้ใช้ MEDIUM
  • หากไม่ส่ง status ให้ใช้ OPEN

BR-04 Delete ​

ไม่บังคับให้ลบ Ticket ในข้อสอบนี้
การยกเลิกให้เปลี่ยน status เป็น CANCELLED

BR-05 Roles and Visibility ​

  • USER เห็น/แก้ไข/เปลี่ยนสถานะได้เฉพาะ ticket ที่มอบหมายให้ตน
  • ADMIN เห็นทุก ticket, สร้าง ticket พร้อมเลือกผู้รับผิดชอบ และเปลี่ยนผู้รับผิดชอบ (reassign) ได้
  • backend บังคับสิทธิ์อยู่แล้ว (เช่น สร้าง = admin เท่านั้น → 403) — frontend แค่ปรับ UI ให้สอดคล้อง

API Contract — Fixly ​

Base URL (be-exam API ที่ deploy ไว้):

text
https://fixly-backend-6pzm.onrender.com/api

เว็บตัวอย่าง (UI reference): https://fixly-frontend-jvxs.onrender.com ใช้ backend ที่ deploy ไว้ตัวนี้ตัวเดียวร่วมกัน — ไม่ต้องรัน be-exam เองในเครื่อง

text

0. Authentication ​

POST /auth/login ​

Request:

json
{
        "username": "demo",
        "password": "password123"
      }

Response:

json
{
        "data": {
          "token": "<jwt>",
          "user": { "id": 2, "username": "demo", "displayName": "Demo User", "role": "USER" }
        }
      }
  • 200 success
  • 401 INVALID_CREDENTIALS

ใช้ user ที่ be-exam seed ไว้ในการ login (เช่น admin/admin123 = ADMIN, demo/demo1234 = USER) role ใช้กำหนดสิทธิ์ใน UI (สร้าง ticket = admin เท่านั้น)

1. Response Shape ​

Single resource ​

json
{
        "data": {}
      }

Collection ​

json
{
        "data": [],
        "meta": {
          "page": 1,
          "limit": 10,
          "total": 24,
          "totalPages": 3
        }
      }

Error ​

json
{
        "error": {
          "code": "VALIDATION_ERROR",
          "message": "Invalid input.",
          "fields": {
            "title": "Title must contain at least 3 characters."
          }
        }
      }

3. Tickets ​

Visibility: ผลลัพธ์ถูกแยกตามผู้ใช้ที่ฝั่ง server — USER ได้เฉพาะ ticket ที่มอบหมายให้ตน, ADMIN ได้ทั้งหมด (frontend ไม่ต้องส่ง filter เพิ่ม)

GET /tickets ​

Query:

Name Type Description
search string title, description, reporterName
categoryId integer filter category
priority enum LOW, MEDIUM, HIGH, URGENT
status enum OPEN, IN_PROGRESS, RESOLVED, CANCELLED
sort string createdAt, priority
order string asc, desc
page integer default 1
limit integer default 10, max 100

GET /tickets/:id ​

  • 200 เมื่อพบ
  • 404 TICKET_NOT_FOUND (รวมกรณี USER ขอดู ticket ที่ไม่ใช่ของตน)

POST /tickets (admin only) ​

Request:

json
{
        "title": "Air conditioner is not cold",
        "description": "The air conditioner in meeting room A is running but not cooling.",
        "reporterName": "Supreme",
        "location": "Meeting Room A",
        "priority": "HIGH",
        "categoryId": 2,
        "assignedToId": 2
      }

Response:

  • 201 success
  • 400 VALIDATION_ERROR (รวมกรณีไม่ส่ง assignedToId)
  • 403 FORBIDDEN (ไม่ใช่ admin)
  • 404 CATEGORY_NOT_FOUND / USER_NOT_FOUND
  • 409 CATEGORY_INACTIVE

PATCH /tickets/:id ​

รองรับ partial update (USER แก้ได้เฉพาะ ticket ของตน; assignedToId reassign ได้เฉพาะ admin)

Response:

  • 200 success
  • 400 validation
  • 403 FORBIDDEN
  • 404 not found / USER_NOT_FOUND
  • 409 category inactive

json

        "status": "IN_PROGRESS"
